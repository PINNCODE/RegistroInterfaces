Inicio

Las metodologías de desarrollo de sistemas de ingeniería de software ofrecen un conjunto de pasos y buenas prácticas para el desarrollo con calidad de un sistema a través del ciclo de vida. Sin embargo, carecen de índices que valoren la madurez del sistema en sus etapas del ciclo de vida desde los requerimientos, código fuente y ejecutable.

No hay una metodología que garantice la confiabilidad de un sistema al 100%, ésta solamente se logra una vez que los sistemas han sido expuestos (compartidos, puestos a prueba, usados, reutilizados), en diferentes niveles de ambientes.

Ante esta problemática se ha desarrollado una guía para la evaluación del nivel de madurez en componentes de software ejecutable, esta guía lleva por nombre Niveles de madurez de componentes para su reutilización basados en la confiabilidad, este documento describe índices que valoran la confiabilidad de un componente a través de la exposición natural que sigue el ciclo de vida de desarrollo de software, independientemente de la metodología de desarrollo y paradigma de programación en la que fue desarrollado el componente.

Niveles de madurez de componentes para su reutilización, basados en la confiabilidad

Para evaluar un componente se debe iniciar en cada aspecto desde el nivel más bajo, si no cumple con un nivel continuar evaluando los niveles superiores.

NOTA: NO es necesario que los niveles se actualicen de manera seriada, es decir, se puede pasar a un nivel más alto desde un nivel inferior.
Niveles de madurez
0 1 2 3 4 5 6 7 8 Todos

Nivel 0
Idea Código Exposición
¿Tienes definidos los parámetros de entrada y salida (requerimientos funcionales) en un diagrama de caja negra? ¿Se tiene la sintaxis de entrada y salida ? ¿Se ha definido el tipo de dato para la entrada y salida del componente?

Nivel 1
Idea Código Exposición
¿Tienes ya la solución en algoritmo o diagrama (de preferencia basado en un libro, artículo, página, repositorio)? ¿Se tiene una relación algoritmo-comentarios-código? ¿Dada cualquier entrada se produce una salida?

Nivel 2
Idea Código Exposición
¿Están definidas las pruebas unitarias a nivel de idea, juguete, promedio y complejas? ¿Hay estructuras que soportan las pruebas unitarias, juguete, promedio y complejas? ¿Pasaste todas las pruebas unitarias?

Nivel 3
Idea Código Exposición
¿Tienes descritos los requerimientos NO funcionales y cómo los has mejorado? (análisis de complejidad) ¿Se han agregado procesos paralelizables, parámetros de E.S. o estructuras de datos para cumplir los requerimientos no funcionales? ¿Cumple el rendimiento con los requerimientos no funcionales?

Nivel 4
Idea Código Exposición
¿Se generó una nueva solución (es solución padre de otras) del componente? (* opcional) ¿Se ha reutilizado código en otro componente? ¿El componente nuevo no presenta fallas en el código?

Nivel 5
Idea Código Exposición
¿Tienes descrito el reúso de tu componente en al menos tres proyectos propios? ¿Es robusto el código ante el reúso en al menos tres proyectos propios? ¿El componente funciona correctamente en al menos tres proyectos propios?

Nivel 6
Idea Código Exposición
¿Tienes descrito el rehúso de tu componente de al menos tres desarrolladores diferentes? ¿Es robusto el código ante el reúso en tres proyectos externos? ¿El componente funciona correctamente en al menos dos desarrollos externos?

Nivel 7
Idea Código Exposición
¿Ha recibido retroalimentación por un desarrollador? ¿El código ha recibido una mejora, modificación o corrección por un desarrollador != al autor? ¿El componente funciona correctamente con las mejoras?

Nivel 8
Idea Código Exposición
¿Tienes descritas cuáles son las referencias a los usuarios y productos que validan este componente? ¿Es robusto el código ante los usuarios, productos y plataformas? ¿Funciona correctamente ante usuarios, productos, plataformas y contingencias?

Estructura del README carnet

La estructura del readme carnet consiste en la unión de los nueve niveles de madurez descritos anteriormente, el propósito del carnet es tener acceso a toda la información relacionada con el componente en cada nivel de madurez que este cumple.

Formato carnet

El formato del readme carnet es .JSON, la razón de utilizar JSON es la facilitar su lectura y escritura, es un formato ligero de intercambio de datos y utiliza convenciones que son familiares para los programadores, en lenguajes como C, C ++, C #, Java, JavaScript, Perl, Python y muchos otros. Estas propiedades hacen de JSON un lenguaje ideal para el intercambio de datos.

Ejemplo nivel de madurez 7

La generación del archivo carnet.json se generara a través de formularios establecidos por cada nivel de madurez, por un lado será visible el formulario con sus campos correspondientes y del otro se visualizara la parte del json que se esta generando a partir de ese formulario.
